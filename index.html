How to interpret a ModForensics result
ModForensics v3.0.1 · Rho-9 Systems · Static analysis platform
The threat score is a triage indicator. It is not a verdict.
ModForensics performs automated static analysis and assigns a weighted numeric score based on the capabilities and indicators it can observe inside the submitted archive. An elevated score means the file contains behaviour that warrants human review. It does not, by itself, establish that the file is malicious, and it never has. A substantial proportion of entirely legitimate, widely distributed modifications score in the elevated bands for ordinary reasons, the most common of which is network access.
The evidence is the product. The number is only the index.
Every score is published alongside the material that produced it: the flagged files, the extracted strings, the indicators of compromise, the class references, the mixin targets and the full decompiled source. That material is provided so that you will read it and reach your own final determination. Quoting a score without reviewing the evidence behind it is a misuse of this tool and is not a finding of any kind.
Detailed guidance · expand as required
01 · What the score actually measures45s

The score is a capped aggregate of weighted heuristics, normalised to a maximum of 100. Each contributing heuristic corresponds to a property that can be observed statically: a matched malware family signature, an indicator of compromise such as a hard-coded webhook endpoint, evidence of obfuscation, or a statistical anomaly such as an unusually high-entropy string.

The engine therefore measures capability and resemblance. It does not measure intent, authorship or outcome. No submitted code is executed at any stage, so no runtime behaviour is ever observed. Two archives that score identically may differ entirely in purpose, and only a human reading the evidence can distinguish between them.

02 · Why a clean mod can legitimately score high45s

The following are normal, expected properties of professionally built modifications. Each one raises the score, because each one is also present in genuine malware. Their presence is a prompt to review, not an accusation.

  • Network access. Update checks, version manifests, distribution platform APIs (CurseForge, Modrinth), content delivery endpoints, server browsers, analytics, crash reporting and Discord Rich Presence all require outbound connections. The engine records outbound endpoints because credential theft uses the identical mechanism. A URL in a mod is a fact to be checked, not proof of anything.
  • Obfuscation and name mangling. Commercial modifications, anti-cheat components, licensed plugins and any project shipped through an optimiser such as ProGuard will present shortened or randomised class names and unreadable control flow. This raises the score because obfuscation frustrates review, not because it is inherently hostile.
  • High-entropy strings. Compressed resources, embedded images, base64 assets, certificates, licence tokens and encryption keys are all statistically indistinguishable from a packed payload at the string level.
  • Reflection, Mixins and bytecode manipulation. The Forge and Fabric ecosystems are built on these mechanisms. Ordinary gameplay modifications routinely redirect, overwrite and inject into game internals as a matter of standard practice.
  • Keyword collision with theft signatures. A modification that reads system properties for a diagnostic report, accesses the clipboard for a coordinate-sharing feature, or resolves the game directory for configuration will overlap textually with signatures derived from information stealers.
  • Bundled and shaded dependencies. A shaded library carries its own indicators into the host archive, and those indicators are attributed to the archive as a whole.

In short: the properties that make a modification powerful are the same properties that make malware possible. The tool cannot separate the two on your behalf.

03 · Score bands and their intended meaning45s
BandRangeIntended operational meaning
Low risk0 to 34No strong indicators surfaced. Review remains advisable for any file obtained outside official distribution channels.
Suspicious35 to 69Indicators are present and require human interpretation. This band is routinely occupied by legitimate networked modifications.
High threat70 to 100Multiple or high-weight indicators surfaced concurrently. Review is required before use, and the archive should not be executed until that review is complete.

These thresholds are deliberately calibrated toward sensitivity. The tool is designed to prefer a false positive that an informed reader dismisses in a few minutes over a false negative that results in a compromised account. Elevated bands should therefore be expected during normal use.

05 · Recommended review procedure45s
  1. Open the Overview tab and read the itemised reasons. The reasons, not the number, are the finding.
  2. Open IOC and examine every endpoint individually. Establish whether each one resolves to a recognised vendor, platform or content delivery domain, or to an anonymous host, a raw paste service, a webhook or a bot API.
  3. Open Inspector and read the call sites in context. Determine what triggers the network call, what data is assembled before it, and whether that data includes credentials, session material, tokens or file contents.
  4. Review Strings and Classes for the paths and identifiers being touched, particularly launcher directories, browser profiles and session files.
  5. Review Mixins for injection targets that are unrelated to the modification's advertised function.
  6. Compare the archive against the author's published source repository and the official distribution listing. Verify that the file you analysed is the file the author actually published.
  7. Record your conclusion and the evidence that supports it. Export the report if the finding needs to be shared.
06 · Known limitations of static analysis45s
  • Analysis is static only. Code is never executed, so behaviour that materialises only at runtime is not observed.
  • Detection is signature and heuristic driven. Novel, targeted or heavily obfuscated payloads may not surface at all.
  • A low score is not a certificate of safety. It indicates only that no strong indicator was surfaced by the current signature set.
  • Decompilation is approximate. Reconstructed source may be incomplete or misleading, particularly where obfuscation is present.
  • Signature sets and weights are revised over time. A result is valid for the version of the engine that produced it and should be re-run after significant updates.
  • The service is provided without warranty of any kind. Responsibility for the final determination, and for any action taken on the basis of it, rests entirely with the person conducting the review.
07 · Responsible conduct when reporting a finding45s

Publicly attributing malicious intent to an author or project on the strength of a score alone is unsupported, damaging and frequently incorrect. Before any public statement:

  • State the specific indicators observed and reproduce the evidence, rather than citing a number.
  • Confirm the finding against the original file from the official distribution channel, since repackaged and re-uploaded archives are a common source of false attribution.
  • Contact the maintainer privately first where the finding is ambiguous.
  • Escalate confirmed compromises to the hosting platform and, where relevant, to r/SusMinecraftMods so that other users can verify independently.
